# ISSUE-010: 一级导航重构 — 评测合并 + 知识阅读扩充

- **Component**: web/index.html, web/app.js, web/style.css

## Background

经过多轮迭代（决策视图下线、踩坑与工具下线），当前一级导航剩余 4 个 Tab：

```
功能对比 | 需求筛选 | 能力雷达 | 延迟分析 | 钱包集成类型
```

其中「功能对比」「能力雷达」「延迟分析」三者数据同源、视角互补，但拆成三个一级 Tab 导致用户在同一服务商的不同维度间频繁跳转。「钱包集成类型」目前只做架构科普，内容单薄，缺少对测试方法论的说明。

## Problem Statement

1. **评测数据碎片化**：同一份评测数据被拆成 3 个一级 Tab，用户无法在一个页面内获得完整画像
2. **知识内容不足**：用户看到测试项名称（如 `create_wallet`、`sign_typed_data`）后缺少"为什么测这个"和"具体测了什么"的解释
3. **一级导航过多**：5 个 Tab 在窄屏下拥挤，信息层级扁平

## Design: 新导航结构

### 一级 Tab（3 个）

```
┌──────────┐  ┌──────────┐  ┌──────────┐
│ 评测总览  │  │ 需求筛选  │  │ 了解更多  │
└──┬───────┘  └──────────┘  └──┬───────┘
   │                            │
   │ 二级子 Tab                  │ 二级子 Tab
   ├─ 功能对比                   ├─ 钱包集成类型
   ├─ 能力雷达                   ├─ 测试项设计思路
   └─ 延迟分析                   └─ 测试项详解
```

### 用户动线

```
评测总览（浏览数据、建立印象）
  → 需求筛选（带着需求缩小范围）
    → 了解更多（补充知识、理解方法论）
```

### 二级子 Tab 交互

- 二级 Tab 栏固定在对应 section 顶部，样式与一级 Tab 做区分（更小、更轻）
- 切换二级 Tab 时不触发一级 Tab 变化，仅替换下方内容区
- URL hash 可选编码：`#overview/radar`、`#learn/test-design`，支持分享直达

## 「评测总览」子 Tab 内容

| 子 Tab | 原 Tab | 内容 | 备注 |
|--------|--------|------|------|
| 功能对比 | 功能对比 | 服务商卡片（通过率 + 基础能力 + DeFi 场景） | 配合 ISSUE-009 融合后的卡片 |
| 能力雷达 | 能力雷达 | 雷达图 + DeFi 热力图 + 场景推荐 | 不变 |
| 延迟分析 | 延迟分析 | 延迟热力图 + 统计卡片 + 免责声明 | 不变 |

## 「了解更多」子 Tab 内容

| 子 Tab | 内容 | 数据来源 |
|--------|------|----------|
| **钱包集成类型** | 现有的架构类型科普（MPC、TEE、Smart Contract 等） | 已有静态内容 |
| **测试项设计思路** | 为什么选择这些测试维度？覆盖了哪些风险面？评分权重如何设定？ | 新增，需撰写 |
| **测试项详解** | 每个 test_id 的具体操作步骤、判定标准（pass/fail 条件）、背后意义 | 可从 YAML 评估文件 + test metadata 提取 |
| **下一步计划** | 候选评测对象调研、OpenClaw 生态钱包 Skill 扫描、未来评测维度扩展方向 | 调研产出，详见 `next-steps.md` |

### 测试项设计思路（内容大纲草案）

```
1. 设计原则
   - 以"开发者集成钱包到 DApp"为核心场景
   - 覆盖完整生命周期：创建 → 签名 → 交易 → 多链 → 治理
   - 优先测试用户可感知的能力（非底层密码学细节）

2. 测试维度与风险映射
   - 钱包核心 (wallet_core): 资产安全基础
   - 应用层 (app): 用户可感知的功能完整度
   - DeFi 对接: 实际业务场景的可行性
   - 基础设施 (infra): 稳定性、文档、SDK 质量
   - 治理合规 (governance): 企业级需求

3. 评分方法
   - 通过率计算规则（排除 not_applicable / inconclusive）
   - 雷达图维度聚合逻辑
   - DeFi 场景 Agent 集成难度评级（即用 / 即用† / 低门槛 / 中等 / 不可行）
```

### 测试项详解（内容结构草案）

每个测试项一个卡片或表格行：

```
┌───────────────────────────────────────────┐
│ T-01 create_wallet          钱包核心       │
│ ──────────────────────────────────────── │
│ 测试内容: 调用 SDK 创建新钱包，验证返回    │
│          有效地址且地址格式符合 EIP-55      │
│ 判定标准: ✅ 返回合法地址 / ❌ 抛出异常    │
│ 为什么重要: 钱包创建是所有后续操作的前提，   │
│           失败意味着集成完全不可用           │
└───────────────────────────────────────────┘
```

### 下一步计划（内容大纲草案）

详细内容见 [`next-steps.md`](./next-steps.md)。页面展示时分三个区块：

```
1. 候选评测对象（OpenClaw 生态）
   - 从 ClawHub 7,300+ Skill 中筛选出具备钱包原语 + 链上执行能力的项目
   - 每个候选列出：名称、能力摘要、支持链、架构类别、ClawHub 原始链接
   - 标注推荐优先级（P0 / P1 / P2）

2. 已排除的项目及原因
   - 列出调研过但不符合 wallet-bench 范围的项目
   - 说明排除理由（如"纯数据查询，无钱包原语"）

3. 未来评测维度扩展方向
   - x402 支付协议能力
   - Agent 经济自主性（autonomous payment）
   - 企业级策略引擎深度评测
```

## 「需求筛选」Tab

保持现有功能不变，仍作为独立一级 Tab。其价值在于：
- 区别于"浏览型"的评测总览，提供"目标驱动型"的交互
- 用户可勾选特定需求（链支持、托管模式、治理能力等）快速定位服务商

- [ ] **AC-010-6**: 撰写「测试项设计思路」内容并实装为子 Tab

### Phase 3: 样式适配

## Anti-Recommendations

- ❌ **不要用下拉菜单替代二级 Tab** — 二级内容是平级关系，Tab 更直观
- ❌ **不要在「了解更多」里放评测数据** — 保持知识内容与动态数据的边界清晰
- ❌ **不要一次性渲染所有子 Tab 内容** — 懒加载，切换时才渲染

## Open Questions

1. 「测试项详解」内容是手写 Markdown 还是从 YAML 评估文件自动提取？前者更灵活，后者更易维护
2. 「了解更多」是否需要增加 FAQ 子 Tab？（如"为什么没有 XX 服务商？""数据多久更新一次？"）
