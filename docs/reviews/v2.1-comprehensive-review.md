# wallet-bench v2.1 全面审查 — 问题清单与优化建议

**审查时间**: 2026-03-09
**审查范围**: Dashboard 全部页面 + 评分系统 + 数据质量 + 代码质量 + 战略缺口
**数据基线**: 12 providers × 66 tests, 3 runs/provider, generated 2026-03-08T17:10

---

## P0 — 必须修复（阻塞可用性或产出错误结果）

### P0-1: cross_chain 维度全 0 — 扭曲雷达图和排名

**影响**: 能力雷达 Tab, 详情页雷达, PlatformScore
**现象**: 12 家 provider 的 cross_chain 雷达维度全部为 0。
- DeFi Matrix 没有 cross_chain 对应的场景条目
- 仅有的 2 个测试 (cross_chain_bridge, arb_atomicity) 全部返回 not_applicable/unsupported
- 雷达图在 cross_chain 方向塌陷为 0，视觉上误导用户以为跨链能力完全不存在

**建议**:
- 方案 A: DeFi Matrix 增加 `cross_chain_bridge` 场景条目（如 Wormhole/Stargate 桥接评级）
- 方案 B: 如果确实无法评测，在雷达图和 Coverage 中移除 cross_chain，改为 4 场景
- 方案 C: 保留但在 UI 标注 "暂无数据" 而非显示 0

### P0-2: fetch 错误处理不足 — 数据加载失败时白屏

**影响**: 所有页面
**现象**: app.js 有 4 个 fetch 调用但只有 1 个 .catch()。如果 public_results.json 或 decision_view.v1.json 加载失败（网络错误/404），Dashboard 白屏无提示。
**建议**: 所有 fetch 加 `.catch()` + 用户友好的错误提示（"数据加载失败，请刷新"）

### P0-3: market 数据只覆盖 6-7 家 WaaS — 5 家 OpenClaw Skill 无市场数据

**影响**: 市场活跃度 Tab
**现象**:
- market_npm.json: 仅 privy, coinbase, crossmint, bnbchain_mcp, moonpay, minara（6 家）
- market_github.json: 加了 okx_onchainos（7 家）
- market_pypi.json: 仅 coinbase（1 家）
- 5 家 OpenClaw Skill (clawlett, para_wallet, universal_trading, polymarket_agent, coinpilot_hyperliquid) 完全没有市场数据

**建议**: 补齐 5 家 OpenClaw Skill 的 GitHub 数据（它们都有 GitHub repo），或在 Tab 中标注 "暂无市场数据"

## P1 — 应该修复（明显的质量问题）

### P1-1: ag01 tool_discovery 阈值过严 — 仅 2/12 通过

**影响**: agent_autonomy 维度评分
**现象**: `_MIN_PUBLIC_METHODS=10` 导致仅 bnbchain_mcp(10) 和 okx_onchainos(10) 通过，其余 10 家（包括 Coinbase、Privy 等强 provider）全部 fail（9 methods）。
**根因**: WalletAdapter 基类有 ~8 个公共方法，子类加 1-2 个就到 9-10。阈值 10 恰好卡在边界。
**建议**: 降到 `_MIN_PUBLIC_METHODS=9`，让主流 adapter 通过 introspection。或改为 "method 数 / 基类 method 数" 比例打分。

### P1-2: 8 个 skip 测试膨胀测试总数但不贡献信息

**影响**: 评分分母, 覆盖率展示
**现象**: session_delegation, webhook_delivery, quota_disclosure, policy_method_scope, rbac, approval_workflow, audit_export, secret_rotation — 这 8 个 "观察项" 在所有 provider 中均为 skip。
**分析**: skip 状态在评分中被排除（不计入 scored 分母），所以不影响分数。但在 UI 中显示 "66 tests" 可能误导用户以为测试了 66 项，实际有效仅 58 项。
**建议**: 在测试总数旁标注 "58 项有效 / 8 项观察"，或在汇总卡片只显示 scored 数量

### P1-3: Clawlett 数据不可信 — placeholder 凭证导致 16 errors

**影响**: 评分排名可信度
**现象**: Clawlett 凭证是 placeholder，16/63 测试 error（"Missing Clawlett script"），得分 7.4%。这不是 Clawlett 的真实能力，而是配置不完整。
**建议**: 在 Dashboard 中给 Clawlett 加 "⚠️ 凭证未配置，结果不代表真实能力" 标注，或从排名中暂时移除

### P1-4: next-steps.md 未提及 para_wallet 和 universal_trading

**影响**: 了解更多 Tab → 下一步计划
**现象**: docs/next-steps.md 缺少 para_wallet 和 universal_trading 的描述
**建议**: 更新 next-steps.md 补全 12 家 provider 信息

### P1-5: 63 vs 66 测试数不一致 — 可能造成评分微偏

**影响**: 数据一致性
**现象**: Clawlett/Minara/Para Wallet 只有 63 tests（缺 attestation, failover_continuity, policy_depth），其余 9 家有 66 tests。
**根因**: runner 的 `_is_not_applicable()` 对这 3 家直接跳过了这些 arch-specific 测试，而不是记为 N/A。
**建议**: runner 应将跳过的测试记为 NOT_APPLICABLE 并计入结果，保证所有 provider 都有 66 条记录

## P2 — 锦上添花（改善体验）

### P2-1: _looks_like_success 仍重复 6 份代码

**影响**: 维护成本
**现象**: d01, m01, x01, t16, t17, t18 各有独立的 `_looks_like_success()` 定义
**建议**: 提取到 `cases/shared/_utils.py` 统一维护

### P2-2: Coverage 徽章缺少无障碍文本

**影响**: 可访问性
**现象**: ●●●○○ 圆点没有 aria-label 或 title 属性，纯视觉表达
**建议**: 加 `aria-label="覆盖 3/5 场景"` + 在圆点旁显示文本 `3/5`

### P2-3: app.js 3557 行 — 单文件过大

**影响**: 可维护性
**现象**: 所有 render 函数、scoring 函数、事件处理都在一个文件
**建议**: 后续可拆分为 scoring.js, render.js, events.js 等模块（非紧急）

### P2-4: URL hash 容错 — 未知 scenario 值无降级

**影响**: 能力雷达 Tab
**现象**: 手动编辑 URL hash 为 `#radar&scenario=typo` 时行为未定义
**建议**: 未知 scenario 值 fallback 到 `all`

### P2-5: 延迟分析 heatmap 数据不均

**影响**: 延迟分析 Tab
**现象**: Clawlett 有 0 条延迟数据，Para Wallet 有 49 条（其余 66 条）。heatmap 可能有空白行。
**建议**: 无延迟数据的 provider 在 heatmap 中标灰色 + "无数据"

### P2-6: 详情页 AI Insights 内容陈旧

**影响**: 详情页
**现象**: AI_INSIGHTS 对象中的文案是 45-test 时代写的，引用的具体数字（如延迟、分数）可能与 66-test 结果不匹配
**建议**: 更新 AI_INSIGHTS 文案适配 v2.1 数据，或加声明 "评语基于早期版本数据，分数请参照实时计算结果"

## P3 — 后续考虑（战略方向）

### P3-1: 缺少 "快速选型" 入口

**现象**: 用户（AI Agent 开发者）需要在 4 个 Tab 间切换才能做出选型决策。缺少一个直接回答 "我要做套利，该用哪个" 的入口。
**建议**: 在首页加 "场景推荐" 快速入口：选场景 → 直接显示 top 3 推荐 + 理由

### P3-2: 无 provider 对比功能

**现象**: 不能选 2-3 个 provider 并排对比
**建议**: 加 checkbox → "对比选中" 按钮 → 并排雷达图 + 差异高亮表

### P3-3: 缺少套利/farming 专项测试

**现象**: Oracle 之前建议补 4 个关键测试：
1. quote→execution 漂移（报价到执行的价差）
2. 并发 nonce/队列冲突
3. 长会话重试稳定性
4. 循环策略的 approve/revoke 卫生
目前未实现。
**建议**: 下个 ISSUE 补充这 4 项，直接服务核心用例

### P3-4: 无历史趋势追踪

**现象**: 只有单次快照数据，无法比较 provider 随时间的改善/退化
**建议**: 引入日期版本化的 results 文件 + 简单的趋势折线图

### P3-5: 评分公式中 cross_chain 维度空转

**现象**: cross_chain 在 PlatformScore 中占据 AppTop2 候选之一的位置，但所有 provider 都是 0。这意味着 AppTop2 永远只从 4 个维度中选，cross_chain 的权重被浪费了。
**建议**: 在补充 cross_chain 数据之前，Coverage 计算和 AppTop2 应排除 cross_chain，避免拉低分数

## 汇总

| 优先级 | 数量 | 代表性问题 |
|--------|------|-----------|
| **P0** | 3 | cross_chain 全 0、fetch 无容错、市场数据缺 5 家 |
| **P1** | 5 | ag01 阈值过严、skip 展示、Clawlett 不可信标注、docs 不全、63 vs 66 |
| **P2** | 6 | 代码重复、无障碍、app.js 过大、hash 容错、heatmap 空白、AI 评语陈旧 |
| **P3** | 5 | 快速选型入口、对比功能、套利专项测试、历史趋势、cross_chain 空转 |

**建议优先级**: P0-1 (cross_chain) → P0-2 (fetch 容错) → P1-1 (ag01 阈值) → P0-3 (市场数据) → P1-3 (Clawlett 标注)
