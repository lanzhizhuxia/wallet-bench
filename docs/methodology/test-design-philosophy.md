# 测试项设计思路

> wallet-bench 为什么选这些测试？覆盖了哪些风险？评分怎么算？

---

## 一、我们在回答什么问题

wallet-bench 的核心受众是 **需要把钱包集成到 DApp / AI Agent 中的开发者和技术决策者**。

所有测试项围绕一个核心问题展开：

> **「这个钱包服务商，我能用它做什么？做起来有多靠谱？」**

我们不关心密码学原语的细节（那是底层库的事），也不做安全审计（那需要专业的安全团队）。我们只关心**开发者在集成过程中会遇到的真实场景**。

## 二、设计原则

### 2.1 覆盖完整生命周期

一次钱包集成不是只调一个 API。从创建钱包到上线运营，开发者需要经历完整的链路：

```
创建钱包 → 签名验证 → 发起交易 → 多链适配 → 费用预估
    → 错误处理 → 并发与限流 → 审计追踪 → 运维恢复
```

测试项沿着这条链路设计，确保每个环节都被覆盖。

### 2.2 以运行时验证为主

大部分测试通过**实际调用 API 并检验返回结果**来判定，而非依赖文档声明或人工检查。

- 说支持创建钱包？调一下，看返回的地址格式对不对
- 说支持签名？签一下，看签名是不是空的
- 说支持多链？查 adapter 声明了几条链

少数无法自动化验证的维度（治理完整度、文档质量等），通过人工 / AI 评估的 YAML 评分补充。

### 2.3 对"测不了"诚实

不同架构的钱包能力边界不同。wallet-bench 用 6 个状态区分"真的不行"和"我们还没测到"：

| 状态 | 含义 | 是否计入评分 |
|------|------|:---:|
| `pass` | 测试通过 | 是 |
| `fail` | 测试失败（服务商能力问题） | 是 |
| `error` | 运行时异常（可能是服务商也可能是基准侧） | 是 |
| `unsupported` | 服务商确认不提供此能力 | 是（扣分） |
| `inconclusive` | 基准侧未接入，无法判定 | 否（降低置信度） |
| `skip` | 行业空白（目前无供应商支持） | 否 |
| `not_applicable` | 测试不适用于该架构/分类 | 否 |

这意味着：如果一个服务商拿到 80% 通过率但置信度偏低，说明有部分测试尚未完成验证——结论需要打折。

### 2.4 按架构分层

钱包服务商的底层架构差异很大：

| 架构类型 | 代表供应商 | 密钥在哪 |
|---------|-----------|---------|
| **本地密钥 (local)** | BNB Chain MPC、Universal Trading、Polymarket Agent | 用户设备上 |
| **TEE / 可信执行环境 (tee)** | （保留扩展） | 服务商安全硬件中 |
| **意图架构 (intent)** | Crossmint、Coinbase AgentKit、OKX OnchainOS、Coinpilot Hyperliquid | 用户提交"意图"，服务商执行 |
| **委托签名 (delegated)** | Privy、MoonPay | 服务商代管密钥，用户授权 |
| **托管 (custodial)** | Minara | 完全由服务商托管 |
| **智能账户 (smart_account)** | Clawlett | AA 合约钱包（Gnosis Safe 等） |
| **MPC (mpc)** | Para Wallet | 多方计算分片密钥 |

不同架构有不同的关注点：本地密钥要验证派生路径的确定性，TEE 要验证安全环境证明，意图架构要验证输入校验。

因此 wallet-bench 除了所有服务商都跑的**通用测试**外，还有按架构分类的**专属测试**。

## 三、评分体系 v2（9 维加权）

> 自 ISSUE-024 起，评分体系从 v1（7 维等权雷达）升级为 v2（9 维加权 + 双评分公式）。

### v2 设计哲学

wallet-bench 的核心用户是 **AI Agent 开发者**，场景是跑链上策略（套利/薅羊毛/挖矿）。因此：

1. **应用场景拆为独立维度** — Swap、DeFi 借贷、跨链、预测市场、永续合约各自独立计分，专精型供应商不被不支持的场景拖低分。
2. **双评分公式** —
   - **FitScore**（场景适配分）：用户选一个目标场景，按 `0.55×场景分 + 0.20×Agent自主性 + 0.15×性能 + 0.10×钱包基础` 排序。
   - **PlatformScore**（平台综合分）：取最强两个场景均值，按 `0.27×A + 0.23×P + 0.20×W + 0.20×AppTop2 + 0.10×E` 全局排行。
3. **治理+安全+运维合并为"企业就绪度"** — 降低非核心维度的噪声权重。
4. **性能独立成维度** — 套利场景对延迟极度敏感。
5. **覆盖度徽章** — `Coverage = 场景分≥40的数量 / 4`（注：`cross_chain` 因行业成熟度不足已从 Coverage 统计中排除），展示广度但不进入分数计算。

### 9 个雷达维度

| 分组 | 维度 key | 中文 | 说明 |
|------|----------|------|------|
| 应用场景 | `swap` | Swap 兑换 | Token 兑换、路由发现、滑点保护 |
| 应用场景 | `defi_lending` | DeFi 借贷 | Aave/Morpho 借贷协议操作 |
| 应用场景 | `cross_chain` | 跨链 | 跨链桥接、状态一致性 |
| 应用场景 | `prediction_market` | 预测市场 | Polymarket 等预测市场操作 |
| 应用场景 | `perps` | 永续合约 | Hyperliquid 等永续合约交易 |
| 基础/加分 | `agent_autonomy` | Agent 自主性 | 工具发现、错误自恢复、多步规划 |
| 基础/加分 | `performance` | 性能 | 并发、延迟、Gas 估算精度 |
| 基础/加分 | `wallet_basics` | 钱包基础 | 创建钱包、签名、转账——基线门槛 |
| 基础/加分 | `enterprise_readiness` | 企业就绪度 | 治理+安全+运维合并 |

## 四、测试维度详解

### 4.1 钱包基础 (wallet_basics)

**关注点**：最基本的钱包操作能否正常工作？

| 测试 | 风险映射 |
|------|---------|
| 创建钱包 | 集成的第一步，失败意味着完全无法使用 |
| 消息签名 | 登录验证、off-chain 授权的基础 |
| 结构化签名 | DeFi 协议交互（Permit、订单签名）的前提 |
| 转账交易 | 资产转移的核心能力 |
| 多链支持 | DApp 是否能跨链部署 |
| 预估手续费 | 用户体验（展示预估 Gas）和风控（防止超额扣费） |
| 交易序号管理 | Nonce 冲突会导致交易卡住或覆盖 |
| 交易确认 | tx_hash 格式不对会导致链上状态无法追踪 |
| ERC-20 转账 | 标准代币转账 calldata 构建与提交能力 |
| 合约写入 | 发送含非空 calldata 的合约调用交易 |
| 签名验证 | ECDSA 恢复 + EIP-1271 合约钱包验签，确保签名可验证 |
| 交易最终性 | 等待链上确认数达到最低要求，确保交易不可回滚 |

**评分**：纯自动化通过率 — `pass / (pass + fail + unsupported) * 100`

### 4.2 权限治理 → 企业就绪度 (enterprise_readiness)

**关注点**：钱包能否满足企业级的风控和审计需求？

| 测试 | 风险映射 |
|------|---------|
| 会话授权 | Agent 是否能在有限范围内自主操作，而非拥有无限权限 |
| 操作审计日志 | 出了问题能否追溯——谁在什么时候做了什么 |
| 审计日志导出 | 能否批量导出审计记录供合规使用 |
| 拒绝交易说明 | 交易被拒时，错误信息是否足以排查——还是只有一个 "failed" |
| 方法级策略控制 | 策略引擎能否限制特定合约方法的调用权限 |
| 角色访问控制 (RBAC) | 是否支持细粒度的角色-权限分离 |
| 审批工作流 | 高风险操作是否支持多签/多人审批流 |

**评分**：混合权重 — 60% 自动化通过率 + 40% YAML 人工评分的"治理完整度"（策略引擎、多签、审批流的覆盖程度）

**现状**：会话授权、方法级策略、RBAC、审批工作流、审计导出目前所有供应商均未支持，作为行业成熟度观察指标保留。

### 4.3 稳定性 → 性能 (performance)

**关注点**：生产环境下的真实负载能否承受？

| 测试 | 风险映射 |
|------|---------|
| 并发操作 | 多用户同时操作时是否会出现竞态条件 |
| 故障恢复 | 传入非法参数后，SDK 是否崩溃、还是优雅报错 |
| 限流恢复 | 触发 API 限流后，能否自动恢复 |
| 重复提交防护 | 同一笔交易提交两次，是否会产生双花 |
| 失败重试机制 | 限流时是否提供 retry-after 等重试指引 |
| 响应时延 SLA | P95 响应时间是否在可接受范围（≤ 10s） |
| 幂等性保障 | 相同参数重复提交是否有去重/防双花机制 |
| 浸泡稳定性 | 持续 N 轮操作后成功率是否维持在 90% 以上 |

**评分**：纯自动化通过率

### 4.4 运维能力 → 企业就绪度 (enterprise_readiness)

**关注点**：上线后的日常维护和灾难恢复能力如何？

通用测试：

| 测试 | 风险映射 |
|------|---------|
| 钱包迁移恢复 | 重启服务后钱包还能不能用 |
| 事件回调通知 | 能否被动接收交易状态变更（而非轮询） |
| 配额透明度 | API 限额是多少？还剩多少？什么时候重置？ |
| 版本兼容性 | Adapter 元数据完整，核心接口 smoke test 通过 |
| 密钥/凭证轮换 | API 密钥是否支持原地轮换而不中断服务 |

架构专属测试：

| 架构 | 测试 | 风险映射 |
|------|------|---------|
| 本地 | 密钥派生路径 | 同一私钥是否总是派生出同一地址 |
| 本地 | 密钥锁定 | 没有正确密钥时是否被阻止操作 |
| 本地 | 备份恢复 | 使用相同密钥能否恢复钱包 |
| TEE | 安全环境证明 | 密钥确实在安全环境中，而非明文存储 |
| TEE | 故障切换恢复 | 服务商宕机重启后，钱包是否仍可访问 |
| TEE | 精细化风控 | 策略引擎是否真的能拦截违规交易 |
| 意图 | 意图格式校验 | 畸形输入是否在本地就被拦截 |
| 意图 | 履约时效 | 异步操作是否在可接受时间内完成 |
| 意图 | 取消操作 | 待处理的操作能否被撤销 |

**评分**：混合权重 — 50% 自动化通过率 + 50% YAML 人工评分的"网络环境"和"文档上手度"均值

### 4.5 应用能力 → 5 场景维度 (swap / defi_lending / cross_chain / prediction_market / perps)

**关注点**：钱包是否内置了 DeFi 操作工具，还是需要开发者自己构建交易？

| 测试 | 对应 DeFi 场景 |
|------|--------------|
| Token Swap | Uniswap V3 兑换 |
| DeFi 操作 | Aave / Morpho 借贷 |
| 永续合约 | Hyperliquid 永续合约交易 |
| 预测市场 | Polymarket 预测市场 |
| 跨链桥接 | 跨链资产转移 |

**注意**：纯基础设施型供应商（如 Privy、Crossmint、BNB Chain MPC、Para Wallet）不提供内置应用层 API，这些测试对其标记为 `not_applicable`，不影响评分。它们的 DeFi 能力通过「DeFi 场景 Agent 集成难度」单独衡量——评估的是"AI Agent 使用该供应商的签名能力自主完成 DeFi 场景的复杂度"（即用 / 低门槛 / 中等 / 不可行）。

**评分**：优先使用 DeFi 场景 Agent 集成难度矩阵的综合评分（0-100 分制）；无矩阵数据时，回退到混合权重（50% 自动化通过率 + 50% YAML 评分）。

## 五、评分聚合方法（v2）

### 5.1 雷达图（9 维）

雷达图的 9 个维度对应上述维度定义，每个维度 0-100 分。

**PlatformScore**（综合分）公式：
```
AppTop2 = (最高场景分 + 第二高场景分) / 2
PlatformScore = 0.27 × Agent自主性 + 0.23 × 性能 + 0.20 × 钱包基础 + 0.20 × AppTop2 + 0.10 × 企业就绪度
```

**FitScore**（场景适配分）公式：
```
FitScore_j = 0.55 × 场景j分 + 0.20 × Agent自主性 + 0.15 × 性能 + 0.10 × 钱包基础
```

**Coverage**（覆盖度）：场景分 ≥ 40 的数量 / 4（`cross_chain` 维度因行业成熟度不足已排除），不进入分数计算。

### 5.2 DeFi 场景 Agent 集成难度

针对 4 个典型 DeFi 场景（Uniswap V3 Swap、Aave/Morpho 借贷、Hyperliquid 永续、Polymarket 预测），从 **AI Agent 自主执行**的视角评估每个供应商的集成难度。

> **为什么不是人类开发者时间？** 传统的"需要 X 天开发"对 Agent 场景意义不大——Agent 生成代码的速度极快，真正的瓶颈在于**架构是否允许 Agent 自主完成**、**交易步骤有多复杂**、**是否依赖外部服务或人工介入**。

#### 评级体系

| 评级 | 中文 | 含义 | 代理分 | 典型特征 |
|------|------|------|:---:|------|
| 🟢 ready_to_use | 即用 | SDK/工具内置该场景，Agent 声明即用 | 100 | 零额外编码，调用一个 action 即完成 |
| 🟢 ready_to_use† | 即用† | 内置能力可用但有限制条件 | 80 | 如仅聚合器（非协议直连）、仅特定链 |
| 🟡 low_barrier | 低门槛 | 需要 calldata 编码但步骤少 | 65 | 有 ABI 就行，1-2 步交易，无外部依赖 |
| 🟠 moderate | 中等 | 多步交易 + 协议专属编码 + 外部依赖 | 40 | 需自定义 EIP-712 编码器、外部 API 等 |
| 🔴 not_feasible | 不可行 | 架构缺少必要能力 | 0 | 无签名、无 calldata、无密钥导出 |

#### 评估维度

每个场景从 4 个维度综合判定：

| 维度 | 说明 | 对 Agent 的影响 |
|------|------|----------------|
| **集成模式** | 声明即用 / calldata 组装 / 协议专属编码 / 不可行 | 决定 Agent 是否需要生成代码 |
| **交易步骤** | 完成该场景需要几步链上交易 | 步骤越多，排序和错误处理越复杂 |
| **外部依赖** | 是否需要调用第三方 API 或使用外部工具 | 增加失败点和延迟 |
| **人工介入** | 是否需要人工确认或手动操作 | Agent 自主性的核心瓶颈 |

支持 3 种权重预设：等权、DeFi 偏重、交易偏重，分别对 4 个场景赋予不同权重，计算加权综合分。

### 5.3 置信度

置信度分为三个档位，综合考量以下因素：

- **接入完整性**：有效测试数 / 总测试数（`inconclusive` 不算有效测试）
- **稳定性**：多次运行结果是否一致（pass↔fail 漂移程度）
- **API 可靠性**：是否频繁触发 429 / timeout

| 置信度 | 供应商 | 含义 |
|--------|--------|------|
| 🟢 高 | Privy、Coinbase AgentKit、MoonPay、BNB Chain MCP | 接入完整，结果稳定，可直接参考 |
| 🟡 中 | Crossmint、Minara、Universal Trading、Polymarket Agent、Coinpilot Hyperliquid | 接入基本完整，偶有不稳定，结论可参考但需关注标注项 |
| 🔴 低 | OKX OnchainOS、Para Wallet、Clawlett | 接入不完整或稳定性差，评分仅供参考 |

置信度低意味着该服务商的评分可能不够准确，用户应参考详情页了解具体哪些测试尚未验证。

## 六、局限性与后续方向

### 当前局限

1. **测试网环境**：所有交易类测试在测试网执行，主网行为可能不同
2. **单次快照**：测试结果是某一时刻的快照，API 变更后可能失效
3. **延迟数据依赖网络环境**：本地测试的延迟不代表通用基准（详见延迟分析免责声明）
4. **应用层测试覆盖有限**：DeFi 场景测试依赖供应商是否暴露了 SDK 接口

### 后续扩展方向

- 增加更多 DeFi 场景（NFT 铸造、DAO 投票等）
- 增加安全维度（重放攻击防护、签名验证）
- 支持更多链（Solana、Tron 等非 EVM 链）
- 引入社区贡献的 adapter，扩大供应商覆盖
