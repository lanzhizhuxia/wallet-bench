# 测试项详解

> 每个测试具体做了什么、怎么判定通过、为什么重要

## 阅读说明

- **测试方式**：`自动` = 通过代码调用 API 实际执行；`混合` = 自动执行 + 人工评估补充；`观察` = 行业成熟度观察，暂无供应商支持
- **测试方式**：`自动` = 通过代码调用 API 实际执行；`混合` = 自动执行 + 人工评估补充；`观察` = 行业成熟度观察，暂无供应商支持
- **适用范围**：`通用` = 所有服务商；`本地` / `TEE` / `意图` = 仅对应架构类型
- **维度归属**：测试所属的 v2 雷达维度（wallet_basics / enterprise_readiness / performance / agent_autonomy 等）

## 一、钱包基础 (wallet_basics)

> v2 维度归属：`wallet_basics`

钱包最核心的能力：创建、签名、交易。这些是所有上层功能的基石。

### T-01 创建钱包 `key_generate`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 调用 `adapter.create_wallet()`，获取返回的钱包地址 |
| **通过条件** | 返回地址匹配 `0x` + 40 位十六进制字符（EIP-55 格式） |
| **失败条件** | 地址为空、格式不合法、或调用抛出异常 |

**为什么重要**：钱包创建是所有后续操作的前提。如果这一步失败，整个集成链路完全不可用。这也是最基本的 smoke test——能创建地址，说明 API 连通性、认证凭证、网络配置都正确。

### T-02 消息签名 `sign_message`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 调用 `adapter.sign_message("hello wallet-bench")`，使用 EIP-191 / personal_sign 标准 |
| **通过条件** | 返回非空签名字符串 |
| **失败条件** | 签名为空 |

**为什么重要**：消息签名是 Web3 登录验证（如 Sign-In with Ethereum）的基础。大量 DApp 用它来证明"这个地址的主人确实在操作"。不支持消息签名意味着无法做最基本的身份验证。

### T-03 结构化签名 `sign_typed_data`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 构造一个标准 EIP-712 结构体（domain: `wallet-bench`, chainId: 97, primaryType: `Test`），调用 `adapter.sign_typed_data()` |
| **通过条件** | 返回非空签名字符串 |
| **失败条件** | 签名为空 |

**为什么重要**：EIP-712 结构化签名是 DeFi 协议的核心需求。Uniswap Permit、OpenSea 的 Seaport 订单、Aave 的授权委托，全部依赖 EIP-712。不支持结构化签名 = 无法与主流 DeFi 协议交互。

### T-04 转账交易 `send_tx`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 构造一笔测试网交易——向 burn 地址 `0x...0001` 转 0.0001 ETH（10^14 wei），调用 `adapter.send_transaction()` |
| **通过条件** | 返回 `tx_hash`；或交易因余额不足被 revert（仍算通过——说明构建+签名+提交链路完整，只是钱包没钱） |
| **失败条件** | 既无 `tx_hash` 也无 revert 标记 |

**为什么重要**：发送交易是区块链钱包的终极目的。这个测试验证了完整的交易链路：构建 → 签名 → 提交到链上。即使交易因余额不足而 revert，也说明整个链路是通的。

### T-05 多链支持 `multi_chain`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 检查 `adapter.chains` 声明的链数量是否 >= 2，并执行一次 `create_wallet()` 验证功能正常 |
| **通过条件** | 声明 >= 2 条链且 `create_wallet` 成功 |
| **失败条件** | 仅声明单条链，或 `create_wallet` 失败 |

**为什么重要**：现代 DApp 通常需要部署在多条链上（Ethereum + Base + Polygon 等）。单链钱包意味着每条链需要不同的集成方案，运维复杂度成倍增长。

### T-10 预估手续费 `preflight_fee`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 构造一笔 0 value 交易，调用 `adapter.estimate_gas()` |
| **通过条件** | 返回非空的 Gas 估算结果 |

**为什么重要**：用户在确认交易前通常需要看到预估的 Gas 费用。没有这个能力，DApp 要么无法展示费用预览，要么需要自己调用链上 RPC 做估算——增加了集成复杂度。

### T-19 交易序号管理 `nonce_management`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 连续发送两笔交易（均为 0 value 到 burn 地址），检查两笔交易是否都成功且 `tx_hash` 不同 |
| **通过条件** | 两笔交易均返回 `tx_hash`，且两个 hash 不相同 |
| **失败条件** | 第二笔交易失败（nonce 冲突）、`tx_hash` 为空、或两个 hash 相同（覆盖） |

**为什么重要**：以太坊的 nonce 机制要求每笔交易有唯一递增的序号。nonce 管理不当会导致交易"卡住"（pending 永远不确认）或"覆盖"（后一笔替换前一笔）。这是高频交易场景的致命问题。

### T-20 交易确认 `tx_confirmation`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 发送一笔交易，验证返回的 `tx_hash` 格式是否为有效的十六进制字符串 (`0x[0-9a-fA-F]+`) |
| **通过条件** | `tx_hash` 非空且格式有效 |
| **失败条件** | `tx_hash` 为空或格式不合法 |

**为什么重要**：`tx_hash` 是链上状态追踪的唯一锚点。格式不正确的 hash 无法在区块浏览器中查询，也无法被下游系统用于确认交易状态，导致"交易发出去了但不知道结果"的黑洞状态。

### T-27 ERC-20 转账 `erc20_transfer`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **维度归属** | `wallet_basics` |
| **具体操作** | 在本地构建 ERC-20 `transfer(address,uint256)` calldata（selector `0xa9059cbb`），向 burn 地址转 1 wei，调用 `adapter.send_transaction()` |
| **通过条件** | 返回 `tx_hash`；或交易因余额/合约不存在而 revert（链路完整即可） |
| **失败条件** | 既无 `tx_hash` 也无 revert 标记 |

**为什么重要**：ERC-20 是最广泛使用的代币标准。如果钱包不能提交含 calldata 的交易，就无法与任何 ERC-20 合约交互，意味着无法操作 USDC、USDT、WETH 等主流代币——这是 DeFi 集成的硬性前提。

### T-28 合约写入 `contract_write`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **维度归属** | `wallet_basics` |
| **具体操作** | 构造一笔含非空 calldata（默认 `0xd09de08a` increment 选择器）的合约调用交易，调用 `adapter.send_transaction()` |
| **通过条件** | 返回 `tx_hash`；或交易触发 revert（说明链路完整，合约层面的失败可接受） |
| **失败条件** | 含 calldata 的交易被拒绝或抛出非 revert 类异常 |

**为什么重要**：发送含 calldata 的交易是与任何智能合约交互的基础——Uniswap、Aave、ENS 注册、NFT 铸造全部依赖这一能力。不支持非空 calldata = 只能转 ETH，无法触达 DeFi 生态。

### T-29 签名验证 `sig_verify`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **维度归属** | `wallet_basics` |
| **具体操作** | 对消息 `"wallet-bench-sig-verify"` 发起签名，然后：(1) EOA：ECDSA ecrecover 恢复 signer 地址；(2) 合约钱包：调用链上 `EIP-1271 isValidSignature()` 验证；若供应商仅能 self-attest（sw_attestation），记录并通过 |
| **通过条件** | recovered signer == wallet address；或 EIP-1271 返回 magic `0x1626ba7e`；或供应商自报告签名者匹配 |
| **失败条件** | 签名为空、recovered address 不匹配、EIP-1271 返回非 magic 值 |

**为什么重要**：签名能力必须是可验证的，而不只是返回一个字符串。Sign-In with Ethereum、订单签名、链上权限委托都依赖第三方能验证签名真实性。如果签名无法被独立验证，整个信任链就断了。

> ⚠️ **sw_attestation 说明**：部分供应商（如 Crossmint）的合约钱包仅能通过 provider 自报告 signer 来证明签名有效性，无法在公链上直接验证。这是架构限制，不是能力缺失，benchmark 对此标记但不扣分。

### T-30 交易最终性 `tx_finality`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **维度归属** | `wallet_basics` |
| **具体操作** | 发送一笔 0 value 交易，获取 `tx_hash` 后，通过 RPC 轮询 `eth_getTransactionReceipt` 直到确认数 ≥ 最低要求（默认 2） |
| **通过条件** | 交易在超时前获得足够确认数，receipt 状态为成功（status=1）或可接受的 revert |
| **失败条件** | 超时前未获得最低确认数，或 receipt 无法获取 |

**为什么重要**：`tx_hash` 只代表交易被广播，不代表上链。确认数不足的交易可能被回滚——对于资金操作，这是不可接受的风险。这个测试验证「提交 → 上链 → 确认」的完整路径是否顺畅。

## 二、权限治理 → 企业就绪度 (enterprise_readiness)

> v2 维度归属：`enterprise_readiness`

企业级钱包需要回答的问题：谁能做什么？做了什么？被拒绝时为什么？

### T-07 会话授权 `session_delegation`

| 项目 | 内容 |
|------|------|
| **测试方式** | 观察 |
| **适用范围** | 通用 |
| **当前状态** | 所有供应商均标记为 `skip`（行业空白） |
| **预期操作** | 创建有时间/金额限制的子会话 → 在范围内操作（应成功）→ 超出范围操作（应失败）→ 撤销会话 |

**为什么重要**：AI Agent 场景下，你不希望给 Agent 无限的钱包权限。理想情况是"每次最多花 0.1 ETH，24 小时后权限自动过期"。这是 Agent 安全性的关键缺口，目前整个行业都还没有好的方案。

### T-13 操作审计日志 `authorization_audit_trace`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 依次执行创建钱包 → 消息签名 → 发送交易，检查每步返回结果中是否包含可审计字段 |
| **检查项** | (1) 钱包地址 + 链 + 时间戳 (2) 签名非空 + hex 格式 + signer 字段 (3) tx_hash + 响应耗时 (4) capabilities 声明完整性 (5) 元数据字段完整性 |

**为什么重要**：当出现"这笔钱转到哪里去了？"的问题时，你需要完整的操作记录链。缺少审计字段意味着事后追溯无从下手。对于合规要求较高的场景（金融、企业财务），这是硬性要求。

### T-22 拒绝交易说明 `denial_reason_quality`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 故意发送一笔不可能成功的交易——向零地址转 10^30 wei（天文数字），观察错误响应的质量 |
| **通过条件** | 错误响应包含结构化字段：`error_code` + `message`（而非仅有一个原始异常字符串） |

**为什么重要**：当交易被拒绝时，开发者需要知道"为什么"。一个好的错误信息应该是 `{code: "INSUFFICIENT_FUNDS", message: "余额不足以支付 gas"}` 而不是 `Error: execution reverted`。这决定了排查问题需要 5 分钟还是 5 小时。

## 三、稳定性 → 性能 (performance)

> v2 维度归属：`performance`

从"能用"到"能在生产环境扛住"的差距。

### T-08 并发操作 `concurrent_ops`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 使用 `asyncio.gather` 并行执行 5 个操作（2x 创建钱包 + 2-3x 消息签名 + 可选 1x 发送交易） |
| **通过条件** | 所有操作均成功，且两次 `create_wallet` 返回不同地址（无竞态重复） |
| **失败条件** | 任一操作失败、返回空结果、或出现重复地址 |

**为什么重要**：真实的 DApp 不是一个用户一个用户排队操作的。并发场景下，内部锁机制、连接池、签名队列是否能正确处理，决定了系统在高峰期是正常运行还是返回随机错误。

### T-09 故障恢复 `failure_recovery`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 执行两个故障注入场景：(1) 向非法地址 `0xinvalid` 发交易 (2) 向 burn 地址转 10^30 wei |

### T-11 限流恢复 `rate_limit_resilience`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |

### T-23 重复提交防护 `idempotent_submit`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 用相同参数发送两笔交易（0 value 到 burn 地址），观察行为 |
| **通过条件** | 第二次被拒绝（优雅报错），或返回相同 `tx_hash`（幂等） |
| **失败条件** | 两次返回不同的 `tx_hash`（产生了两笔链上交易 = 双花风险） |

**为什么重要**：网络抖动导致的重复提交是常见场景。如果钱包服务商不做幂等处理，一笔转账可能变成两笔——这就是"双花"。对于资金操作，这是不可接受的。

### T-24 失败重试机制 `retry_backoff`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **具体操作** | 连续执行 10 次 `sign_message` 触发限流，检查限流响应是否包含 `retry-after` / `backoff` 信息，等 2 秒后再试 |
| **通过条件** | 场景一：未触发限流（连续调用稳定）；场景二：触发限流且提供了重试指引，冷却后恢复 |
| **失败条件** | 触发限流但不提供重试信息，或冷却后仍无法恢复 |

**为什么重要**：好的 API 在限流时应该告诉调用方"多久后可以重试"（`Retry-After: 5`），而不是让调用方盲猜。对于自动化的 AI Agent，没有 backoff 指引 = 要么疯狂重试浪费资源，要么无限等待浪费时间。

### T-35 响应时延 SLA `timeout_sla`

| 项目 | 内容 |

### T-36 幂等性保障 `idempotency_key`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **维度归属** | `performance` |

## 五b、Agent 可用性 → Agent 自主性 (agent_autonomy)

> v2 维度归属：`agent_autonomy`（a01-a05, ag01-ag06）

AI Agent 集成质量测试：返回结构完整性、错误可机读性、响应确定性、调用成本、多步恢复。

### A-01 输出结构质量 `schema_quality`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **维度归属** | `agent_autonomy` |
| **具体操作** | 执行 create_wallet / sign_message / send_transaction 三种操作，验证每个返回对象的字段完整性：地址 0x 前缀、链标识、签名格式、tx_hash 格式、错误时 error_code + message 字段存在 |
| **通过条件** | 全部验证项均通过 |
| **失败条件** | 关键字段缺失、格式不匹配或值为空 |

**为什么重要**：AI Agent 不是人，它靠机器可读的字段而非自然语言描述来作出决策。如果 address 字段缺失或签名格式不一致， Agent 要么崩溃，要么需要额外的提示词工程才能继续。

### A-02 机器可读错误处理 `machine_errors`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **维度归属** | `agent_autonomy` |
| **具体操作** | 注入两种错误：(1) `sign_message("")` 空消息签名 (2) `send_transaction` 到非法地址。检查错误响应是否包含结构化字段（error_code / message）而非只有原始异常字符串 |
| **通过条件** | 两种错误均返回可机读错误信息 |
| **失败条件** | 错误响应为空或仅有原始异常消息 |

**为什么重要**：Agent 的错误自恢复能力取决于错误信息的质量。一个 `{"code": "INVALID_ADDRESS", "message": "..."}`  让 Agent 知道该换什么参数重试；而一个 `Error: execution failed` 让 Agent 盲目地重试或放弃。

### A-03 响应确定性 `deterministic_response`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **维度归属** | `agent_autonomy` |
| **具体操作** | 用相同参数连续调用 3 次 `sign_message`，比较：(1) 签名是否均非空 (2) 签名长度是否一致 (3) signer 字段存在性是否一致 (4) signer 字段值是否一致 |
| **通过条件** | 4 项检查均通过 |
| **失败条件** | 签名为空、长度浮动或 signer 字段不一致 |

**为什么重要**：Agent 的多步工作流依赖一致的输出格式。如果同样的调用返回结构每次都不同，Agent 的解析逻辑就不稳定——有时个字段存在，有时不存在，这让 Agent 需要写复杂的防御性代码。

| 项目 | 内容 |

### A-05 多步恢复 `multi_step_recovery`

| 项目 | 内容 |
|------|------|
| **测试方式** | 自动 |
| **适用范围** | 通用 |
| **维度归属** | `agent_autonomy` |
| **具体操作** | 3 步流程：(1) create_wallet 成功 (2) send_transaction 到非法地址（应优雅失败） (3) send_transaction 到正常地址（应成功——验证第 2 步错误没有污染状态） |
| **通过条件** | 第 3 步在第 2 步失败后仍成功 |

**为什么重要**：Agent 的工作流必然包含错误。关键问题是错误属于“环境失败”还是“adapter 状态污染”。一个子步骤失败应该只影响那一步，而不是让整个流程从此崩溃。
